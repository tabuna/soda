# Бэклог: задачи по аудиту архитектуры

## Что даст успех и что — нет

**Задачи эпиков A–F сами по себе не делают продукт «как Laravel» для конечного пользователя.** Laravel — полноценный фреймворк с единым входом, соглашениями, генераторами и богатым CLI. Soda — узкий CLI-инструмент; в аудите «как Laravel» означало **предсказуемый код, короткие классы, fluent там, где повтор, один источник правды для правил** — то есть **DX для контрибьюторов и сопровождения**, а не копирование Laravel целиком.

| Цель | Обеспечивают задачи |
|------|----------------------|
| Меньше багов и дрейфа правил, быстрее добавлять правила | A, B, частично D |
| Проще тестировать и эволюционировать ядро | B, C |
| Удобнее жить пользователю `soda quality` (ошибки, вывод, CI) | **G** (отдельно) |
| «Ощущение полировки» как у хороших CLI | G + стабильный контракт JSON + README |

**Итог:** выполнение A–F — *необходимая* база для устойчивости; **отличный пользовательский опыт** требует эпика **G** (и по желанию пунктов из H). Без G успех по архитектуре есть, «восторг пользователя» — не гарантирован.

Краткие задачи с **критериями приёмки (DoD)**. Порядок внутри эпика — рекомендуемый; между эпиками можно параллелить, если нет зависимости.

## Эпик A — Единый каталог правил

### A1. Ввести `RuleCatalog` (или эквивалент) с полным списком правил

**Цель:** один перечень id правил с полями: секция, label, severity, comparison, default limit.

**Приёмка:**

- [ ] Есть класс (например `RuleCatalog`) с методом, возвращающим все определения правил (id → метаданные + default threshold).
- [ ] Количество правил в каталоге **совпадает** с множеством ключей в `QualityConfig::DEFAULT_RULES` (или задокументировано исключение).
- [ ] `phpstan analyse src` без новых ошибок; существующие тесты зелёные.

---

### A2. ~~Генерация штрафов score из каталога~~ (снято)

Агрегированный quality score и штрафы за нарушения **удалены**; итог прогона — только список `Violation`.

---

### A3. Синхронизация `RuleMetadata` с каталогом

**Цель:** label/severity/comparison для отчёта не расходятся с `*RuleDefaults`.

**Приёмка:**

- [ ] Либо `RuleMetadata::default()` строится из каталога, либо есть **один** unit-тест, который сравнивает ключи и поля каталога с тем, что отдаёт `RuleMetadata` (падение при рассинхроне).
- [ ] `ReportFormatter` / существующие тесты форматирования проходят.

---

### A4. Генерация `RuleSections` из каталога

**Цель:** секции `structural` / `complexity` / … не дублируют списки вручную.

**Приёмка:**

- [ ] `RuleSections::sections()` и `sectionNames()` формируются из каталога (или общего источника).
- [ ] `RuleSectionsTest` (или новый тест) проверяет стабильность структуры `soda.json` (все секции и ключи на месте).

**Зависимости:** A1.

---

## Эпик B — Извлечение метрик файла

### B1. Выделить класс анализа одного PHP-файла

**Цель:** `QualityAnalyser` не содержит детальной логики parse + visitors + breathing merge.

**Приёмка:**

- [ ] Новый класс (например `PhpFileQualityExtractor` / `AnalysePhpFileForQuality`) с публичным методом вида `extract(string $filePath): FileQualitySlice` (имя и тип DTO — на усмотрение PR), возвращающим те же данные, что сейчас собирает `analyseFile()`.
- [ ] `QualityAnalyser::analyseFile()` делегирует в этот класс (или заменён вызовом без дублирования тела).
- [ ] Поведение end-to-end не меняется: существующие тесты + при необходимости один новый unit на изолированный вызов extractor с fixture-файлом.

---

### B2. Централизовать список AST-visitor’ов для quality

**Цель:** порядок и набор visitor’ов в одном месте.

**Приёмка:**

- [ ] Отдельный метод/класс (например `QualityAstPipeline::attachTo(NodeTraverser $t): QualityAstContext`), без копипасты списка из `QualityAnalyser`.
- [ ] Добавление нового visitor’а в коде — **одно** место регистрации (плюс сам класс visitor).
- [ ] Тесты зелёные.

**Зависимости:** B1 (логично в одном PR или сразу после).

---

## Эпик C — CLI и тестируемость

### C1. Подмена `QualityAnalyser` в `QualityCommand`

**Цель:** тестировать команду без файловой системы / полного прогона.

**Приёмка:**

- [ ] У `QualityCommand` есть способ передать `QualityAnalyser` (конструктор с опциональным аргументом и дефолт `new QualityAnalyser()`, либо статическая фабрика — без нарушения Illuminate Console).
- [ ] `QualityCommandTest` использует stub/mock/fake analyser и проверяет хотя бы: успешный exit code при заданном результате или вызов `analyse` с ожидаемыми аргументами.

---

### C2. Интеграционный тест команды `quality`

**Цель:** дымовой тест реального CLI на временной директории.

**Приёмка:**

- [ ] Тест создаёт temp dir, кладёт минимальный `.php` и при необходимости `soda.json`, запускает приложение/команду так же, как e2e (или через `Application::call`).
- [ ] Проверяется код возврата и/или фрагмент вывода.
- [ ] Тест в suite `unit` или отдельной пометке; не flaky на CI.

**Зависимости:** по желанию C1 (упрощается моками, но не обязательно).

---

## Эпик D — Конфигурация правил

### D1. Отключение отдельного правила через конфиг

**Цель:** не выставлять «фиктивные» пороги для выключения проверки.

**Приёмка:**

- [ ] В `soda.json` поддержан согласованный механизм: например `null` для ключа правила **или** `enabled: false` на уровне правила (выбрать один стиль и описать в README/docs).
- [ ] Отключённое правило **не** создаёт `Violation` (или явно задокументировано иначе).
- [ ] Unit-тест на `QualityConfig` / merge + тест на движок с отключённым правилом.

---

### D2. (Опционально) Backed enum `RuleId`

**Цель:** меньше опечаток в PHP-коде.

**Приёмка:**

- [ ] Enum с case на каждое правило из каталога; маппинг `RuleId::fromString` / `->value` для JSON.
- [ ] Публичный JSON-формат ключей **не ломается**.
- [ ] PHPStan level проекта без новых baseline-ignore.

**Зависимости:** A1 (желательно после каталога).

---

## Эпик E — Документация и DX контрибьютора

### E1. Документ «Как добавить правило»

**Цель:** новый разработчик проходит чеклист без чтения всего аудита.

**Приёмка:**

- [ ] Файл `docs/ADDING_A_QUALITY_RULE.md` (или аналог) с нумерованными шагами: метрика (visitor), порог в конфиге, checker, регистрация, тесты, метаданные отчёта.
- [ ] В `README.md` или `docs/ARCHITECTURE_AUDIT.md` — одна ссылка на этот файл.

---

### E2. Ограничение размера checker-классов (гайдлайн)

**Цель:** удерживать читаемость.

**Приёмка:**

- [ ] В `ADDING_A_QUALITY_RULE.md` или CONTRIBUTING: ориентир «новый checker до ~150 строк; иначе разбить».
- [ ] Не обязательно рефакторить старые классы в той же задаче.

---

## Эпик G — Пользовательский опыт (UX) и «полировка» CLI

Без этого блока архитектурные эпики **не гарантируют** отличный опыт у того, кто просто запускает `soda quality`.

### G1. Ошибки конфигурации и окружения — с контекстом

**Цель:** пользователь понимает *что* сломалось и *где*, без чтения исходников.

**Приёмка:**

- [ ] Сообщения `ConfigException` / ошибки чтения JSON содержат **путь к файлу** и по возможности **ключ/секцию** (`rules.structural.max_method_length`).
- [ ] Невалидный JSON — одна строка с причиной парсера + путь (уже частично есть — проверить полноту).
- [ ] Пример некорректного фрагмента в `docs/` или README в разделе troubleshooting (опционально, но желательно).

---

### G2. Понятный итог прогона

**Цель:** за 3–5 секунд понять: есть ли нарушения и что править.

**Приёмка:**

- [ ] В конце текстового отчёта: явная строка **[OK]/[FAIL]** при наличии/отсутствии нарушений (см. README).
- [ ] При FAIL: число issues в одной строке.
- [ ] Тест или снимок вывода на fixture, чтобы регрессия текста была видна (по желанию — минимальный assert в e2e/unit).

---

### G3. Справка команды и примеры для копипаста

**Цель:** снижение трения при первом запуске и в CI.

**Приёмка:**

- [ ] `soda quality --help`: описания опций соответствуют реальному поведению (`--config`, `--report-json`, `--exclude`, …).
- [ ] В README — блок **Quick start** (2–4 команды) + **CI** (фрагмент GitHub Actions или generic shell).
- [ ] Ссылка на формат `soda.json` (существующий пример или `soda.json` в репо).

---

### G4. Стабильность и описание JSON-отчёта

**Цель:** интеграторы CI и IDE не ломаются молча.

**Приёмка:**

- [ ] В `docs/` короткая схема полей `--report-json` (верхний уровень: `schema_version`, `metrics`, `violations`) + версионирование при breaking change (`schema_version`).
- [ ] При изменении структуры JSON в мажоре — запись в CHANGELOG.

---

## Эпик H — Долгосрок / по желанию

### H1. События жизненного цикла анализа

**Приёмка:** задокументированы события; есть минимум один подписчик в тесте или no-op в core; не ломает публичный API без мажорной версии.

### H2. Параллельная обработка файлов

**Приёмка:** флаг CLI; корректный aggregate результат; тест на идентичность с последовательным режиме на малом наборе файлов.

### H3. Минимальный `SodaApplication` / биндинги в контейнере

**Приёмка:** `QualityCommand` получает зависимости через контейнер (или фасад), без обязательного `new` внутри `handle()`; документировано для расширений.

---

## Сводка: маппинг на цели

| Эпик | Кому помогает | Связь с «как Laravel» |
|------|----------------|------------------------|
| A–B | Контрибьюторы | Один источник правды, предсказуемые изменения |
| C–D | Контрибьюторы + продакт | Тесты, гибкий конфиг |
| E | Контрибьюторы | Документированный путь, как у хороших OSS |
| **G** | **Пользователи CLI** | Ближе к «приятному» DX в духе `artisan` (ясность, не магия) |
| H | Платформа / масштаб | Не обязательно для «успеха v1» |
